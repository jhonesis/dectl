## Research Questions

### RQ-001: CLI framework in Rust

**Context**: The entire developer experience of dec depends on the CLI being ergonomic, discoverable, and easy to extend with subcommands. The framework choice affects compile times, binary size, and how much boilerplate each new command requires.

**Options Evaluated**:

| Option | Pros | Cons |
|--------|------|------|
| `clap` v4 (derive API) | Industry standard, rich `--help` output, shell completions, well documented, mature ecosystem | Adds ~1.5MB to binary, longer compile times |
| `argh` | Tiny, fast compile, Google-backed | Limited features, poor `--help` formatting, no shell completions |
| `structopt` | Good ergonomics | Deprecated — merged into `clap` v3+ |
| Manual `std::env::args` | Zero dependencies | Enormous maintenance burden, no help generation |

**Decision**: `clap` v4 with derive API.

**Rationale**: dec's primary users include AI coding environments that parse `--help` output to understand available commands. `clap` produces the best structured help text and supports `--json` output patterns cleanly. The binary size penalty is acceptable given the 20MB limit in the constitution. Shell completions are a valuable future feature that `clap` enables at no extra cost.

### RQ-002: SQLite interface for persistent memory

**Context**: Memory is the most critical persistence layer in dec. The library must be reliable, support transactions, and work without runtime dependencies in the final binary (no shared `.so` files).

**Options Evaluated**:

| Option | Pros | Cons |
|--------|------|------|
| `rusqlite` | Bundles SQLite statically, zero runtime deps, simple synchronous API, battle-tested | No async support (not needed here) |
| `sqlx` | Async, compile-time query checking, multi-database | Requires async runtime (Tokio), overkill for CLI, adds significant compile time |
| `diesel` | ORM with migrations built in | Heavy, complex setup, async support incomplete |
| Flat files (TOML/JSON) | No dependency, human-readable | No query capability, poor performance at scale, no transactions |

**Decision**: `rusqlite` with bundled SQLite feature flag (`bundled`).

**Rationale**: dec is a synchronous CLI tool — async adds complexity with zero benefit. `rusqlite` with `bundled` compiles SQLite directly into the binary, guaranteeing no runtime dependency. Transactions protect memory integrity during writes (aligns with constitution reliability requirement). Simple enough that the full memory module can be implemented in ~200 lines of Rust.

### RQ-003: YAML parsing for workflows

**Context**: Workflows are defined in YAML by developers and potentially generated by AI models. The parser must handle standard YAML gracefully, give useful error messages on malformed files, and integrate well with Rust's type system via `serde`.

**Options Evaluated**:

| Option | Pros | Cons |
|--------|------|------|
| `serde_yaml` | Serde integration, widely used, deserializes directly into typed structs | Based on `yaml-rust` which had maintenance gaps; v0.9 rewrote internals |
| `yaml-rust2` | Community-maintained fork, actively maintained | Lower-level, requires manual parsing into structs |
| `toml` (alternative format) | First-class serde support, Rust-native feel | YAML is more readable for multi-step workflows; conflicts with user expectations |

**Decision**: `serde_yaml` v0.9+.

**Rationale**: Direct deserialization into typed Rust structs (`WorkflowStep`, `StepType`) is essential for reliable workflow execution. The rewrite in v0.9 resolved prior maintenance concerns. YAML stays as the format — it's what developers expect for declarative workflow definitions and what AI models write naturally.

### RQ-004: .gitignore parsing for project scan

**Context**: `dectl project scan` must respect `.gitignore` rules (REQ-005). Implementing gitignore pattern matching correctly is non-trivial — patterns include negations, directory-specific rules, and glob syntax.

**Options Evaluated**:

| Option | Pros | Cons |
|--------|------|------|
| `ignore` crate (by BurntSushi) | Used by `ripgrep`, battle-tested, handles nested `.gitignore`, respects `.git/info/exclude` | Slightly larger dependency |
| `gitignore` crate | Lightweight | Less maintained, doesn't handle nested ignores well |
| Custom implementation | Zero deps | Gitignore spec is complex; high risk of subtle bugs |

**Decision**: `ignore` crate.

**Rationale**: Gitignore pattern matching has well-known edge cases (negation order, directory trailing slashes, nested files). The `ignore` crate is the reference implementation used by `ripgrep` — correctness is guaranteed. The dependency is justified by the complexity it replaces. Using it means `dectl project scan` behaves identically to tools developers already trust.

### RQ-005: Workflow action execution

**Context**: Steps of type `action` in workflows execute system commands. The execution model determines security, portability, and how output is captured and reported.

**Options Evaluated**:

| Option | Pros | Cons |
|--------|------|------|
| `std::process::Command` with explicit arg splitting | No shell injection risk, cross-platform, output captured cleanly | Developer must write commands as array (`cmd: ["git", "status"]`) |
| Shell passthrough (`sh -c "command"`) | Familiar syntax, supports pipes and redirects | Shell injection risk if workflow YAML is untrusted, platform-specific (`sh` unavailable on Windows) |
| Embedded scripting (Lua, Rhai) | Full scripting power | Significant complexity, larger binary, unexpected attack surface |

**Decision**: `std::process::Command` with shell passthrough as an explicit opt-in.

**Rationale**: Default execution uses `std::process::Command` with the command split into args — safe and portable. For cases where a developer genuinely needs shell features (pipes, redirects), they can opt in explicitly with `shell: true` in the workflow step. This keeps the secure path as the default while not blocking legitimate use cases. The constitution's one-time confirmation requirement for `action` steps applies regardless of execution mode.

### RQ-006: Full-text search engine for memory queries

**Context**: Memory entries grow unbounded as the project evolves. Simple `LIKE '%query%'` does not scale — it's slow, ignores ranking, and misses partial matches. The search engine must be fast, local, have zero network dependencies, and run inside the existing SQLite database.

**Options Evaluated**:

| Option | Pros | Cons |
|--------|------|------|
| SQLite FTS5 (built-in via `bundled-fts5` feature) | Zero extra deps, integrated with existing SQLite, ranking (bm25), tokenizers, no separate index | Requires switching from `bundled` to `bundled-fts5` feature |
| Tantivy (Rust full-text engine) | Modern, fast, BM25 ranking | Heavy dependency (~15 crate deps), separate index file, async API |
| Embeddings via pgvector / OpenAI | Semantic search, modern | Requires network, cloud dependence, violates local-first + model-agnostic constitution |
| Manual LIKE + application-level ranking | No deps | Slow on large datasets, no ranking, hacky |

**Decision**: SQLite FTS5 with `bundled-fts5` feature flag.

**Rationale**: FTS5 is already inside SQLite — switching from `bundled` to `bundled-fts5` adds zero bytes to the binary. The FTS5 virtual table sits alongside the existing `memories` table, kept in sync via triggers. Ranking via `bm25()` gives relevance-ordered results. The tokenizer (unicode61) handles natural language gracefully. No network, no extra binary size, no external index — perfectly aligned with the constitution's local-first requirement.

### RQ-007: Query language design for `dectl memory query`

**Context**: Power users need to filter memory by multiple fields simultaneously (type, tags, date range, text). Passing all filters as CLI flags quickly becomes unwieldy. A dedicated query language strikes the balance between expressiveness and simplicity.

**Options Evaluated**:

| Option | Pros | Cons |
|--------|------|------|
| Custom mini-language with tokenizer + parser + SQL builder | Full control, predictable SQL, no injection risk, self-documenting | Requires writing ~300 lines of parser code |
| Pass-through SQL queries | Flexible, no parser code | SQL injection risk (user input directly to SQLite), breaks model-agnostic goal (requires SQL knowledge) |
| Reuse CLI flags (`--type`, `--tag`, `--from`, `--to`) | Familiar clap pattern | Combinatorial explosion, no complex expressions, limited to AND-only queries |
| Third-party query language (Datalog, PRQL, EdgeQL) | Established syntax | Heavy deps, learning curve, overkill for key-value field queries |

**Decision**: Custom field query language with tokenizer → recursive descent parser → SQL builder.

**Rationale**: The language is intentionally limited to field queries (`type:research`, `tags:rust,cli`, `from:2026-01-01`, `is:starred`) — not a general SQL replacement. The tokenizer splits input into tokens, the parser builds an AST limited to conjunctions and noun phrases, and the SQL builder produces parameterized queries with zero injection risk. The full implementation is ~569 lines with 13 unit tests — maintainable and auditable. The syntax is designed to be discoverable by AI models and humans alike.

### RQ-008: Tag taxonomy for memory classification

**Context**: Tags are free-form strings in the current schema (`tags TEXT` in memories). This leads to tag fragmentation ("rust", "Rust", "rustlang") and makes aggregation meaningless. A controlled vocabulary improves search and enables analytics.

**Options Evaluated**:

| Option | Pros | Cons |
|--------|------|------|
| Tag taxonomy table + migration seed (9 standard tags) | Controlled vocabulary, enables future analytics, backward-compatible | Migration required; existing tags remain as-is but new additions are validated |
| Enforce via application-level validation only | No migration, simple | No database constraint, no analytics value |
| Allow any tag + normalized form | Flexible, user-friendly | Tag normalization is lossy (Rust→rust), no curated list for discovery |
| No taxonomy at all | Zero work | Status quo — tag fragmentation worsens over time |

**Decision**: `tag_taxonomy` table with 9 seed tags + migration v4.

**Rationale**: The taxonomy is a reference table, not a constraint — existing free-form tags remain untouched. New additions go through `dectl memory add --tags` which validates against the taxonomy and warns on unknown tags (but doesn't reject them). The 9 seed tags (`rust`, `cli`, `architecture`, `decision`, `workflow`, `agent`, `memory`, `sdd`, `config`) cover the project's domains. The table enables future features like tag suggestions, analytics by category, and memory organized by domain.

### RQ-009: Memory type categorization

**Context**: Memory entries serve multiple purposes (research notes, decisions, code output, session summaries). A flat "everything is a note" model makes filtering hard. Adding a `type` column enables structured categorization.

**Options Evaluated**:

| Option | Pros | Cons |
|--------|------|------|
| `type TEXT CHECK IN (...)` with 7 predefined types | Schema-level validation, backward-compatible (default 'note'), self-documenting | Migration required; existing entries default to 'note' |
| Enum in Rust only | No migration, flexible | No DB-level validation, no query-by-type via SQL |
| Tags instead of type column | No new column, extensible | Tags are many-to-many; type is one-to-one; different semantics |
| Discard and let user organize manually | Zero work | No discoverability, no filterability |

**Decision**: `type TEXT NOT NULL DEFAULT 'note' CHECK(type IN ('note','decision','research','code','session','agent','task'))` — migration v2.

**Rationale**: The type column is a simple enum enforced at the database level. Existing entries get `'note'` as default. The 7 types correspond to actual memory usage patterns observed in the project. The `--type` flag on `dectl memory add` validates against the same enum. During `dectl session end`, decisions are captured as `type='decision'`. Agent auto-link uses `type='research'` for researcher output and `type='note'` for other agents. The CHECK constraint is cheap and prevents data corruption.

## Platform Support: Windows

### Current Status

**Partial support** — binary compiles for Windows but has several runtime incompatibilities.

### Known Issues

#### 1. Shell Execution (Critical)

| Location | Issue |
|----------|-------|
| `src/workflow/runner.rs:107` | Uses `Command::new("sh").args(["-c", ...])` — fails on Windows |

**Impact**: Workflows with `shell: true` cannot execute on Windows.

**Workaround**: Use array-style commands without shell features.

---

#### 2. Environment Variables

| Location | Issue | Fix Available |
|----------|-------|----------------|
| `src/core/config.rs:64` | Uses `HOME` env var directly | `dirs::home_dir()` |
| `src/core/config.rs:69` | Uses `HOME` env var directly | `dirs::home_dir()` |
| `src/memory/db.rs:46` | Uses `HOME` env var directly | `dirs::home_dir()` |
| `src/memory/edit.rs:15-17` | Falls back to `vi` if EDITOR not set | Need Windows fallback |

**Impact**: Config and memory fail to initialize on Windows because HOME is typically not set.

**Workaround**: Set `HOME` manually or use `%USERPROFILE%`.

---

#### 3. Default Editor Fallback

| Location | Issue |
|----------|-------|
| `src/memory/edit.rs:17` | Falls back to `vi` which doesn't exist on Windows |

**Impact**: `dectl memory edit` fails if no editor is configured.

**Workaround**: Set `EDITOR` or `VISUAL` environment variable.

---

#### 4. Path Handling

All path operations use `std::path::Path` which is cross-platform. No hardcoded `/` separators found — good.
