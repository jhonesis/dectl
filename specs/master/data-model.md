## SQLite Entities (`~/.dectl/memory.db`)

### `memories`

The core memory table. Every entry stored via `dectl memory add` lives here.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | INTEGER PRIMARY KEY | вң… | Auto-increment. Used in `dectl memory show <id>` |
| `content` | TEXT | вң… | Markdown-supported content of the memory entry |
| `tags` | TEXT | вқҢ | Comma-separated tags. Nullable. e.g. `"architecture,auth"` |
| `project` | TEXT | вқҢ | Project name at time of creation. Nullable (global memory if absent) |
| `type` | TEXT | вң… | Entry category. One of: `note`, `decision`, `context`, `research`, `incident`, `code-snippet`. Default: `'note'` |
| `created_at` | TEXT | вң… | ISO 8601 timestamp. e.g. `"2026-06-02T14:32:00Z"` |
| `updated_at` | TEXT | вң… | ISO 8601 timestamp. Set on create and when content is edited |
| `deleted_at` | TEXT | вқҢ | ISO 8601 timestamp. Set on soft-delete. `NULL` = active entry |

**Indexes**:
- `idx_memories_project` on `project` вҖ” speeds up project-scoped queries
- `idx_memories_created` on `created_at DESC` вҖ” speeds up chronological listing

**Notes**:
- `content` is stored as raw Markdown. Rendering is the responsibility of the consuming tool.
- `tags` stored as comma-separated TEXT (not a separate table) for simplicity. Parsed as `Vec<String>` on read.
- A memory with `project = NULL` is considered global and returned in all projects.
- Soft-delete: `dectl memory delete <id>` sets `deleted_at`, `--hard` removes the row permanently.
- All queries filter `WHERE deleted_at IS NULL` by default.

---

### `memories_fts`

FTS5 virtual table for full-text search. Auto-synced via triggers on `memories`. External content table вҖ” no manual inserts needed.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `rowid` | INTEGER | вң… | Matches `memories.id` |
| `content` | TEXT | вң… | Full-text indexed content |
| `tags` | TEXT | вқҢ | Included but marked UNINDEXED (only kept for trigger sync) |

**Triggers**:
- `memories_ai` вҖ” AFTER INSERT: inserts new row into FTS index
- `memories_ad` вҖ” AFTER DELETE: removes row from FTS index
- `memories_au` вҖ” AFTER UPDATE: deletes old row, inserts new row

**Notes**:
- Search uses FTS5 ranking (`ORDER BY rank`) with fallback to `LIKE` if FTS5 fails.
- Query terms are joined with `AND` for multi-term search.
- The FTS table is populated with existing memories on migration v2 via `INSERT INTO memories_fts(rowid, content, tags) SELECT id, content, tags FROM memories`.

### `agent_outputs`

Tracks the link between agent executions and their resulting memory entries. Created in migration v3.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | INTEGER PRIMARY KEY | вң… | Auto-increment |
| `agent_type` | TEXT | вң… | Agent name. e.g. `"researcher"`, `"coder"` |
| `task_id` | TEXT | вқҢ | Unique task identifier. e.g. `"T119"` |
| `task_description` | TEXT | вқҢ | Human-readable task description |
| `output_file` | TEXT | вқҢ | Path to agent output artifact in `.dec/agent-output/` |
| `memory_id` | INTEGER | вқҢ | Foreign key вҶ’ `memories.id`. `ON DELETE SET NULL` |
| `created_at` | TEXT | вң… | ISO 8601 timestamp. Default: `datetime('now')` |

**Indexes**:
- `idx_agent_outputs_agent` on `agent_type`
- `idx_agent_outputs_task` on `task_id`

**Notes**:
- Created automatically when an agent completes successfully.
- `memory_id` links to the auto-inserted memory summary. Can be `NULL` if the memory entry was deleted.

Catalog of predefined tags and their categories. Created in migration v4.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | INTEGER PRIMARY KEY | вң… | Auto-increment |
| `name` | TEXT UNIQUE | вң… | Tag name. e.g. `"decision"`, `"high-impact"` |
| `description` | TEXT | вқҢ | Human-readable description of the tag |
| `category` | TEXT | вң… | Tag category. One of: `type`, `meta`, `priority`, `source`. Default: `'general'` |
| `created_at` | TEXT | вң… | ISO 8601 timestamp. Default: `datetime('now')` |

**Indexes**:
- `idx_tag_taxonomy_category` on `category`

**Seed data** (9 tags inserted on migration v4):

|------|----------|-------------|

| `context` | type | Project context and requirements |
| `research` | type | Research findings and analysis |

### `embeddings` *(future вҖ” not yet implemented)*

Stores vector embeddings for semantic search. One row per memory entry.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | INTEGER PRIMARY KEY | вң… | Auto-increment |
| `memory_id` | INTEGER | вң… | Foreign key вҶ’ `memories.id`. Cascade delete |
| `model` | TEXT | вң… | Embedding model used. e.g. `"nomic-embed-text"` |
| `vector` | BLOB | вң… | Raw float32 array serialized as binary |
| `created_at` | TEXT | вң… | ISO 8601 timestamp |

**Indexes**:
- `idx_embeddings_memory_id` on `memory_id`

**Notes**:
- SQLite does not have a native vector type. `vector` is stored as BLOB (serialized `Vec<f32>`).
- Semantic search in Phase 2 loads vectors into memory and computes cosine similarity in Rust вҖ” no vector DB required at this scale.
- If the embedding model changes, old embeddings are invalidated. `dectl memory embed-all --model <name>` regenerates them.

## Relationships

```
memories 1 в”Җв”Җв”Җв”Җ 0..1 embeddings
   (memory_id FK, cascade delete)

memories 1 в”Җв”Җв”Җв”Җ 1..1 memories_fts
   (rowid = id, content-sync via triggers)

memories 1 в”Җв”Җв”Җв”Җ 0..1 agent_outputs
   (memory_id FK, ON DELETE SET NULL)
```

**Notes**:
- A memory entry exists independently of its embedding. Deleting a memory cascades to its embedding.
- `memories_fts` is an external content FTS5 table вҖ” its rowid is always the same as `memories.id`. Sync is automatic via triggers.
- `agent_outputs` links an agent execution to its auto-generated memory summary. `ON DELETE SET NULL` avoids orphaned FKs.
- `tag_taxonomy` is independent вҖ” no FK relationships to other tables. Tags are referenced by name from `memories.tags`.

## Flat File Entities

### `trust.toml` (`~/.dectl/trust.toml`)

Tracks which workflows have been explicitly trusted by the user for execution of `action` steps.

```toml
[trusted]
# key = "<project_path>/<workflow_name>"
# value = ISO 8601 timestamp of when trust was granted

"/home/user/projects/myapp/design_architecture" = "2026-06-02T10:00:00Z"
"/home/user/projects/myapp/setup_project" = "2026-06-02T11:30:00Z"
```

**Rules**:
- Trust is scoped to project path + workflow name. The same workflow name in a different project requires separate trust.
- Trust is never granted automatically. Always requires explicit user confirmation (Y/n prompt).
- Revoking trust = deleting the entry from this file.

## Migration Strategy

- SQLite schema migrations run automatically on `memory.db` open via a `migrations` table that tracks applied versions.
- Migrations are append-only вҖ” never modify or drop existing migrations.
- Each migration is idempotent (`IF NOT EXISTS`) вҖ” safe to re-run.
